架构约束分析

根据需求推导出的系统形态约束

🌐 必须是 Web 后端

为什么：

  • ✅ 跨工作室派单 → 需要中心化服务
  • ✅ 多端实时同步 → WebSocket 推送
  • ✅ 数据汇总/盈亏统计 → 服务端计算
  • ✅ 权限隔离 → 服务端鉴权
  • ✅ 管理员远程访问 → 无需本地部署
💻 陪玩端必须有本地客户端

为什么：

  • ✅ 路由器限流控制 → 需要本地网络权限
  • ✅ 远程开关机 → 需要系统级权限
  • ✅ 状态计时（娱乐/接单）→ 本地追踪
  • ✅ 订单弹窗推送 → 桌面通知
  • ✅ 隐藏运行/低资源占用 → 后台服务

架构方案对比

满足上述约束的三种可行方案

A

Web 全栈 + 轻量 Agent

后端：Node.js / Python（REST + WebSocket）

前端：React / Vue（管理端 + 客服端纯浏览器）

陪玩端：Electron 桌面应用（内含 Web UI + 本地控制模块）

优点

  • 一套 Web 代码复用
  • Electron 可调用系统 API
  • 跨平台

缺点

  • Electron 体积大（~150MB）
  • 内存占用较高
  • 路由器控制需要另写 native 模块
B

Web 后端 + Go Agent（推荐）

后端：Go / Node.js（REST + WebSocket）

前端：React / Vue（管理端 + 客服端纯浏览器）

陪玩端：Go 编译的轻量 agent（~10MB，系统服务 + 本地 Web UI）

优点

  • Agent 体积小、低资源占用
  • Go 天然适合系统/网络编程
  • 单文件部署，静默运行

缺点

  • Agent UI 需额外处理（WebView 或本地 HTTP）
  • Go 团队招聘相对小众
C

Web 全栈 + C# Agent（仅 Windows）

后端：.NET / Node.js（REST + WebSocket）

前端：React / Vue（管理端 + 客服端浏览器）

陪玩端：C# WPF/WinForms 桌面应用 + WebView2

优点

  • Windows 原生，系统控制能力强
  • .NET 生态成熟
  • 可直接调用 Windows API 限速

缺点

  • 仅 Windows，无法跨平台
  • 客户端更新分发较麻烦
