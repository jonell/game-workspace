## 1. 用户故事地图

### 角色定义

| 角色 | 说明 | 现有系统对应 |
|------|------|-------------|
| **OWNER (老板)** | 最高权限，全局管理 | User.role='OWNER' |
| **ADMIN (店长)** | 日常运营管理，管理下属陪玩 | User.role='ADMIN' |
| **CS (客服)** | 派单操作，可查看但不可管理黑名单 | User.role='CS' |
| **COMPANION (陪玩)** | 终端用户，Agent运行在陪玩PC上 | User.role='COMPANION' |
| **SYSTEM (系统)** | 自动化采集、下发、杀进程 | Go Agent + NestJS Server |

### 故事地图（按优先级排列）

#### P0 — 必须（MVP 核心闭环）

| US-002 | SYSTEM | 作为系统，我需要定时（默认5分钟）重新采集进程列表并上报，同时在工作/娱乐模式切换时立刻触发采集 | 定时任务准时执行；模式切换事件触发即时采集，延迟 < 3s | US-001 |
| US-003 | ADMIN/OWNER | 作为管理员，我需要在管理端查看所有陪玩终端上报的进程列表，并能筛选/搜索 | 列表展示陪玩名、进程名、PID、路径、上报时间；支持按陪玩/进程名搜索 | US-001 |
| US-004 | ADMIN/OWNER | 作为管理员，我需要创建黑名单规则，命中规则的进程将在陪玩终端被杀死 | 支持从"已上报进程"中选择或手动输入；规则启用后立即生效 | US-003 |

### 2.5 异常流程

```
异常1: Agent 上报进程时 WebSocket 断开
  → Agent 本地缓存最近一次采集结果
  → WebSocket 重连后批量补报（带时间戳，服务端按时间戳去重）
  → 最多缓存 3 次采集结果（防内存溢出）

异常2: 管理端下发规则时目标 Agent 离线
  → 服务端将规则标记为 "PENDING_SYNC"
  → Agent 上线后通过 WebSocket 握手时主动拉取（或在 heartbeat 响应中带回）
  → 下发状态在 Agent ACK 后更新为 "SYNCED"

异常3: 杀进程操作失败（权限不足 / 进程已退出 / 系统保护）
  → Agent 记录 Win32 错误码
  → 第一次失败后 30s 自动重试 1 次
  → 仍失败则上报 FAILED + 错误原因，不再重试
  → 管理端日志显示红色 "失败" 标签 + 错误详情 Tooltip

异常4: 管理员误加操作系统进程（如 explorer.exe）
  → 服务端规则保存时进行"高危进程"校验
  → 若匹配到预置的系统关键进程列表，弹窗二次确认
  → Agent 端也有本地系统进程白名单做最后兜底

异常5: 微信多实例 / 多子进程
  → Agent 内置微信家族白名单：WeChat.exe, WeChatApp.exe, WeChatWeb.exe, WeChatPlayer.exe
  → 进程路径包含 "Tencent\WeChat" 的全部放过
  → 白名单在 Agent 端本地执行，不依赖服务端（防下发延迟）

异常6: .lnk 快捷方式指向无效路径
  → 拖入管理端时，前端通过 FileReader 读取 .lnk 二进制解析目标路径
  → 若解析失败，弹窗提示"无法解析该快捷方式，请手动输入进程名或路径"
  → 若目标路径指向不存在文件，标记规则为"路径待验证"（黄色警告）

异常7: 全屏游戏场景下弹出通知
  → Agent 检测前台窗口是否为全屏（窗口尺寸 == 屏幕分辨率 且 无标题栏）
  → 全屏时通知静默入队，不弹窗
  → 退出全屏后（检测到窗口变化）再弹出累积通知（最多保留3条）
  → 超过3条则只弹一条"有N条待处理的进程管控通知"
```

- `(companionId, createdAt DESC)` — 按陪玩查日志
- `(ruleId, createdAt DESC)` — 按规则查日志
- `(killStatus, createdAt)` — 失败日志查询
- `(processName, companionId, triggeredAt)` — 顽固进程检测（同一进程名+陪玩+短时间多次）

## 5. 边界条件清单

### 5.1 同一进程多次上报

| 场景 | 处理方案 |
|------|---------|
| 同一 Agent 同一进程名+同一 PID 短时间内多次上报 | 服务端按 `(companionId, processName, pid, reportedAt)` 去重；1分钟内同一 key 只保留最新一条，旧记录标记为 `DUPLICATE` |
| 同一进程 PID 变化（进程重启） | PID 不同则视为不同记录，正常存储 |
| 同一批次重复上报（Agent 断线重连补报） | 使用 `snapshotId` 批次日志去重；同一 `snapshotId` 内的重复不插入 |

### 5.2 进程名变更

| 场景 | 处理方案 |
|------|---------|
| 用户手动重命名 exe 文件（如 cheat.exe → notcheat.exe） | Agent 上报的是实际运行时的进程名，新名称自然上报；若黑名单规则使用通配符匹配（如 `*cheat*`），仍会命中 |
| 规则使用精确路径匹配 | 精确路径会因重命名而失效；建议优先用签名+通配符组合匹配 |
| 建议 | 管理端提示用户："精确路径匹配对文件重命名无效，建议使用签名匹配或通配符" |

### 5.3 循环杀

| 场景 | 处理方案 |
|------|---------|
| 进程被杀死后自动重启（守护进程 / 双进程互相守护） | Agent 维护"顽固进程计数器"：同一 `processName` 在 5 分钟内被杀 > 3 次 → 标记为 `PERSISTENT` |
| 标记为顽固进程后 | 不再自动查杀，改为：① 上报告警事件至管理端 ② 管理端收到"顽固进程告警"通知 ③ 管理员手动决策（忽略/强制关机/远程协助） |
| 告警信息 | 包含进程名、路径、被杀次数、时间窗口、可能原因（双进程守护/自启动服务） |

### 5.4 多同名进程

| 场景 | 处理方案 |
|------|---------|
| 同一终端运行多个同名进程（如多个 chrome.exe） | Agent 逐一检查每个匹配实例；对每个实例都执行杀进程；通知展示为"即将关闭 chrome.exe (共3个实例)" |
| 杀进程时部分成功部分失败 | 日志中每个 PID 独立记录；汇总显示为 `2/3 成功` |
| 微信多个子进程 | 触发白名单过滤，Agent 扫描到 WeChat*.exe 时跳过，不杀 |

### 5.5 .lnk 指向无效路径

| 场景 | 处理方案 |
|------|---------|
| .lnk 指向的目标文件已被删除/移动 | 前端解析 .lnk 后，对提取的路径进行"存在性校验"（不实际访问文件系统，仅验证路径格式是否合法） |
| 校验失败 | 弹窗提示："⚠️ 该快捷方式指向的路径可能无效：D:\old\app.exe。是否仍要添加？（后续将以进程名+签名匹配，而非绝对路径）" |
| 用户确认后 | 规则存储为 `WILDCARD_NAME` 类型，匹配值使用从 .lnk 中解析出的进程名 |
| 技术实现 | 见 9.1 节"拖拽交互推荐方案" |

### 5.6 微信多子进程白名单

| 场景 | 处理方案 |
|------|---------|
| WeChat.exe 主进程 | Agent 白名单 `byName: ["WeChat.exe"]`，直接跳过 |
| WeChatApp.exe / WeChatPlayer.exe 等子进程 | 白名单覆盖所有已知微信子进程名 |
| 微信目录下的其他 exe（如更新程序） | `byPath: ["*\\Tencent\\WeChat\\*"]` 路径通配，目录下所有 exe 均放过 |
| 未来微信新增子进程 | Agent 配置支持热更新白名单；管理端可下发白名单补丁 |
| 兜底 | 若匹配到路径包含 `Tencent\WeChat` 的进程，即使不在 `byName` 列表中也放过 |

### 5.7 管理员误加系统进程

| 场景 | 处理方案 |
|------|---------|
| 管理员在管理端手动添加 `explorer.exe` 或 `svchost.exe` 到黑名单 | **服务端保存校验**：规则保存时，若匹配值命中预置的系统高危进程列表，弹窗二次确认："⚠️ 该规则将影响到操作系统关键进程，可能导致系统不稳定。确定继续？" |
| 管理员确认后仍然添加 | 规则仍会被保存（给管理员最高权限），但 **Agent 端有最终兜底**：Agent 的本地系统白名单优先于服务端下发的黑名单，系统关键进程不会被杀死 |
| Agent 端拒绝杀系统进程 | Agent 上报 `FAILED` + 原因："进程被系统白名单保护，拒绝执行" |

### 5.8 全屏游戏场景

| 场景 | 处理方案 |
|------|---------|
| 陪玩在全屏游戏（如英雄联盟）中，Agent 检测到黑名单进程 | 全屏检测逻辑：`GetForegroundWindow` → `GetWindowRect` → 比较窗口尺寸与当前显示器分辨率；若完全覆盖且窗口无 WS_CAPTION 样式 → 判定全屏 |
| 通知处理 | 不弹窗，通知入队；退出全屏后弹出（最多延迟5分钟）；若5分钟内未退出全屏 → 在游戏结束后推一条汇总通知 |
| 杀进程是否延迟 | 是。全屏时杀进程操作也入队，退出全屏后执行；原因：全屏游戏时杀进程（尤其是 kill 作弊器）可能导致游戏崩溃/封号风险 |
| 紧急情况 | 管理端可标记某规则为 `URGENT`（紧急），紧急规则在全屏时仍然弹窗通知但使用更小更低调的样式 |

## 9. 对需求原文的追问与推荐方案

### 9.1 拖拽交互：浏览器 vs Electron → **推荐: HTML5 DnD + FileReader (方案A)**

| 维度 | 方案A: HTML5 DnD + FileReader | 方案B: Electron shell API |
|------|------------------------------|--------------------------|
| **原理** | 浏览器原生拖拽API，通过 `FileReader.readAsArrayBuffer()` 读取 `.lnk` 二进制，前端解析 ShellLink 格式提取目标路径 | Electron 主进程 `shell.readShortcutLink()` 直接读取，通过 IPC 返回结果 |
| **优点** | ① 无需 IPC 通信，纯前端实现 ② 同时支持 Web 管理端和 Electron 客户端 ③ 不依赖 Node.js 环境 | ① 解析准确率高（Windows API） ② 代码量少 ③ 天然支持所有 .lnk 特性 |
| **缺点** | ① 需要自行实现 ShellLink 二进制解析（约200行 JS） ② 对复杂 .lnk（环境变量、相对路径）解析可能不完整 | ① 仅 Electron 客户端可用，Web 管理端需降级 ② 增加 IPC 通道复杂度 |
| **推荐** | ✅ **方案A** | - |

**推荐理由**:
1. 项目现有管理端是 **React Web** 应用（`apps/web`），同时也有 Electron 客户端（`apps/companion-electron`）。方案A 可同时支持两端。
2. 对于方案A的 ShellLink 解析，可以使用成熟的开源库如 `lnk-parser` 或 `windows-shortcuts`（npm 包），无需从零实现。
3. 降级策略：若浏览器不支持 DnD 或解析失败 → 引导用户手动复制路径粘贴。

**实现建议**:
```typescript
// 前端拖拽处理伪代码
const handleDrop = async (e: DragEvent) => {
  const file = e.dataTransfer.files[0];
  if (file.name.endsWith('.lnk')) {
    const buffer = await file.arrayBuffer();
    const targetPath = parseShellLink(buffer); // 使用 lnk-parser 库
    if (targetPath) {
      setRuleForm({ matchValue: targetPath });
    } else {
      showWarning('无法解析该快捷方式，请手动输入');
    }
  } else if (file.name.endsWith('.exe')) {
    setRuleForm({ matchValue: file.name });
  }
};
```

### 9.2 倒计时秒数：3/5/10s → **推荐: 可配置默认 5s (方案A)**

| 维度 | 方案A: 可配置默认 5s | 方案B: 固定 5s | 方案C: 按规则差异化 |
|------|---------------------|----------------|-------------------|
| **实现** | 1个系统配置项 `kill_countdown_seconds`，默认5，OWNER 可修改为 3~30 | 硬编码 5s | 每条黑名单规则可单独设置倒计时秒数 |
| **优点** | 灵活可调，适应不同场景 | 实现最简单 | 精细化控制：重要进程短倒计时，非关键进程长倒计时 |
| **缺点** | 需要一个全局配置管理 | 无法适配不同场景 | 复杂度高，管理员需要额外决策 |
| **推荐** | ✅ **方案A** | - | - |

**推荐理由**:
1. 5秒是用户体验研究的黄金平衡点：足够用户看到通知并做反应，也不至于太长让用户等待。
2. 可配置性保证了灵活性（如内测阶段可设置为 10s，正式运营后改为 3s）。
3. 方案C（按规则差异化）可作为 **P2 后续迭代**，当前 MVP 阶段方案A 足够。

**配置项设计**:
```
系统设置 > 进程管控:
  杀进程倒计时: [___5___] 秒  (范围: 3~30s, 默认5s)
  全屏延迟处理: [___300__] 秒  (范围: 60~600s, 默认300s)
  顽固进程阈值: [___3___] 次/5分钟
```

### 9.3 OS 进程定义：路径+签名+微软证书组合 → **推荐: 三层过滤 (确认需求方案)**

这个需求原文的方案已经非常合理，我在此基础上补充细化为三层过滤：

| 过滤层 | 规则 | 示例 |
|-------|------|------|
| **Layer 1: 路径白名单** | 路径以 `C:\Windows\` 或 `C:\Program Files\WindowsApps\` 开头 | `C:\Windows\System32\svchost.exe` |
| **Layer 2: 签名 + 微软证书** | 数字签名有效 且 签名者匹配 `Microsoft*` 或 `Microsoft Windows*` | `dwm.exe` 签名者 = `Microsoft Windows Publisher` |
| **Layer 3: 知名系统进程名** | 预置系统进程名列表（参考 4.5 节白名单） | `services.exe`, `lsass.exe`, `winlogon.exe` |

**判定逻辑**: 三层中**任意一层命中** → 标记为 `isSystemProcess = true` → 不上报或上报时标记但管理端显示"系统进程"标签。

**关键补充**: 签名验证必须验证**证书链到受信任的根证书**，而非仅检查有无签名。使用 Win32 `WinVerifyTrust` API 进行完整验证。

### 9.4 采集频率：启动+5分钟定时+模式切换触发 → **推荐: 方案A（确认需求方案）+ 动态调整**

| 触发事件 | 频率 | 策略 |
|---------|------|------|
| Agent 启动 | 启动后 30s 内 | 首次全量采集 |
| 定时采集 | 默认每 5 分钟 | 全量采集 + 增量对比（仅上报变化） |
| 模式切换 (接单↔娱乐) | 切换后 3s 内触发 | 即时全量采集（因为模式切换意味着使用场景变化） |
| 动态降频 | 当连续 3 次采集无变化 → 降为 10 分钟/次 | 节省带宽和服务器资源 |
| 动态升频 | 当检测到进程数量突变（±30%） → 立即触发一次 | 感知可能的异常行为 |

**增量上报策略**（性能优化）:
```go
// Agent 端维护上一次快照的哈希表
type ProcessSnapshot struct {
    Processes []ProcessInfo
    Hash      string // MD5(processName+pid for each process)
}

// 上报时比较哈希
if currentSnapshot.Hash == lastReportedHash {
    log.Println("No process changes, skip report")
    return
}
```

### 9.5 跨电脑同名不同路径：进程名+签名匹配而非绝对路径 → **推荐: 多模式匹配体系 (方案A)**

**核心问题**: 同一应用在不同电脑上安装路径不同（如 `C:\Program Files\App\` vs `D:\Software\App\`），绝对路径匹配会失效。

**推荐方案: 分层匹配模式**

| 匹配模式 | 匹配逻辑 | 适用场景 | 误杀风险 |
|---------|---------|---------|:------:|
| **SIGNER_MATCH** (签名匹配) | 提取进程的数字签名者名称 + 证书指纹，匹配签名 | 已知发行商的应用（知名软件） | 🟢 低 |
| **WILDCARD_NAME** (进程名通配) | `*cheat*`, `hack*.exe` | 已知可疑进程名模式 | 🟡 中 |
| **PATH_CONTAINS** (路径包含) | 路径含 `*cheat*`，正则为 `cheat` | 已知隐藏在特定目录 | 🟡 中 |
| **EXACT_NAME** (精确进程名) | `cheat.exe` | 确定进程名但路径未知 | 🔴 较高 |
| **COMBO** (组合匹配) | 进程名 + 签名约束（如进程名 `obs*.exe` 且 签名 `NoSignature`） | 需精准控制的场景 | 🟢 低 |

**推荐**: Agent 默认使用 **COMBO 模式** — 进程名匹配（通配）+ 签名约束，两者都满足才触发。

**服务端规则存储**: 见 4.2 节 `matchType` 字段设计，支持以上所有模式。

**示例规则**:
```
规则: 禁止未签名录屏软件
matchType: COMBO
matchValue: obs*.exe, bandicam*.exe, fraps*.exe
signerConstraint: NoSignature   ← 关键：只杀未签名的录屏软件，有签名的合法版本放过
```
